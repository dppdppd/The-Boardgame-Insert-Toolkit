# 2026-05-05

## Add sliding lid support

### Accomplished
- Added `LID_TYPE` with `LID_CAP`, `LID_INSET`, and `LID_SLIDING`, preserving legacy inset behavior when the new key is omitted.
- Added configurable sliding-lid open side via `LID_SLIDE_SIDE` and patterned-lid frame width via `LID_FRAME_WIDTH`.
- Implemented printable sliding rails with a horizontal bearing shelf and 45-degree top wedge, open-top slide entry, configurable open side, stop wall, and matching lid bevel.
- Added a 45-degree detent on the opening-side wall with a matching underside groove in the lid.
- Updated non-solid sliding lids to use a perimeter frame and the same wedge/chamfer envelope as solid sliding lids.
- Fixed related square-cavity top interior corner chamfer behavior and rail clipping issues found during closeup rendering.
- Added `tests/v4/scad/test_lid_sliding.scad` using a boxed poker-deck cavity sized from common tuck-box dimensions plus clearance.
- Updated README, parameter guidance, and rendering guidance, including the rule that changed tests should also generate renders.
- Bumped BIT to `4.0.6`, committed `e187a3b feat(lid): add sliding lid`, and pushed `master` to GitHub.

### Key Discoveries
- Sliding-lid geometry needs visual closeups or high-resolution renders; CSG-only checks missed subtle rail, corner, and groove defects.
- The slide opening must remain top-open, not a covered slot, while the side wedges still need to extend flush to the opening wall face.
- Common boxed poker-deck/tuck-box dimensions cluster around `2.625" x 3.625" x 0.75"`; the example cavity now adds clearance beyond that packed-deck envelope.

### What Didn't Work
- Early rail profiles either left the wedge cut short or made the lid stop at the inner wall, leaving no shelf for the lid to rest on.
- STL-import closeup clipping was unreliable for this geometry; source-level clipping and preview renders gave better diagnostics.

### Next
- Continue using render closeups for sliding-lid fit issues before relying on standard views.

## Release version-locked BIT 4.0.8

### Accomplished
- Fixed sliding-lid panel sizing so rail-side clearance accounts for rail width plus `2 * G_TOLERANCE` on each rail side, stop-side clearance uses `G_TOLERANCE`, and the lid is elevated by tolerance in closed visualization.
- Added a front cutaway sliding-lid clearance regression test and reran the full CSG suite successfully.
- Expanded patterned-lid frame behavior so `LID_FRAME_WIDTH` applies to all patterned non-solid lid types, with the default matching wall thickness.
- Increased default `CHAMFER_N` to `0.6`.
- Added the first physical-validation slice and created backlog follow-ups for validation, divider integration, known chamfer gaps, and subcomponent nesting features.
- Built version-lock packaging: `scripts/package-release.sh`, shared version helpers, patched pre-commit versioning, and shipped `release/lib/boardgame_insert_toolkit_lib.4.0.8.scad`.
- Updated README and shipped starter/example files so user designs include the most recent full-version library filename.
- Committed `db73c59 feat: release 4.0.8`, pushed `master`, pushed tag `v4.0.8`, and published the GitHub release with the versioned library asset.
- Added an ignore rule for `*.lock`; this remains uncommitted pending housekeeping.

### Key Discoveries
- Release packaging must be idempotent: rerunning a package command against a matching full-version library should refresh includes without advancing the version.
- Tests should continue targeting the moving development library file while shipped designs target immutable full-version files.
- Explicit `--minor` packaging must still win when the working library was already patch-stamped before release packaging.

### What Didn't Work
- The first package-script draft bumped on every run, which would have created accidental patch releases during verification.

### Next
- Commit or intentionally skip the `.gitignore` rule for `*.lock`.
- Continue the physical-validation backlog from the current IN-PROGRESS task.
