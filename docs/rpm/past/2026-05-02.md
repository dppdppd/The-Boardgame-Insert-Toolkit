## Key Discoveries
- The `tests/v3-baseline/` deny rule (`Edit/Write` in `.claude/settings.json`) is interpreted by the harness to also cover destructive Bash operations like `rm` against that path. The deny isn't tool-list-specific — it scopes the path.
- The harness scopes me strictly to the trusted repo. `cd ../<other-repo>` and any read/write outside the working tree are denied as "scope escalation". Cross-repo work needs a fresh session inside the other repo (or an explicit added permission rule).
- A pre-commit hook that mutates files works cleanly when it `git add`s the file as part of the hook — the new contents are included in the commit being made, no chicken-and-egg.
- Pre-push hook can't easily make a NEW commit included in the current push (push spec is determined before the hook runs); this is why I went with pre-commit instead for the version-stamping use case.
- `sed -i.bak ... && rm -f file.bak` is the portable in-place edit pattern that works on both GNU and BSD sed.
